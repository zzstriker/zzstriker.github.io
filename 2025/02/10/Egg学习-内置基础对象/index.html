Egg学习-内置基础对象

在 Egg.js 中，提供了多个内置的基础对象，这些对象贯穿整个应用的开发过程，更方便地处理请求、响应、配置、日志等任务。

内置基础对象

在 Egg.js 中，内置的基础对象是从 Koa 继承而来的 4 个对象（Application，Context，Request，Response）以及框架扩展的其他一些对象（Controller，Service，Helper，Config，Logger）。

Application

Application 是全局应用对象，一个应用只会实例化一个 Application 对象。它在应用启动时被创建，贯穿整个应用的生命周期。

主要作用

  1. 配置管理：应用的全局配置可以通过 app.config 访问。
  2. 插件管理：通过 app 对象可以访问所有已加载的插件。
  3. 生命周期管理：app 提供了 beforeStart、ready 等生命周期钩子，方便在应用启动前后执行逻辑。
  4. 全局数据共享：可以在 app 上挂载全局数据或方法，供其他模块使用。

Context

Context 是请求级别的对象，每次请求都会创建一个新的 Context 对象。封装了当前请求的上下文信息，包括请求参数、响应方法等。

主要作用

  1. 请求处理：通过 ctx 可以访问请求参数、请求头等信息。
  2. 响应处理：通过 ctx 可以设置响应状态码、响应头、响应体等。
  3. 中间件传递数据：ctx 可以在中间件之间传递数据。

Request 和 Response 对象

Request 和 Response 分别是 Context 对象的两个属性，分别封装了 HTTP 请求和响应的相关信息。

Request

Request 是一个请求级别的对象，继承自 Koa.Request。封装了 Node.js 原生的 HTTP Request 对象，提供了一系列辅助方法获取 HTTP 请求常用参数。

常用属性/方法

  1. request.method：获取请求方法（GET、POST 等）。
  2. request.url：获取请求 URL。
  3. request.headers：获取请求头。
  4. request.body：获取请求体（需要配合 bodyParser 插件使用）。

Response

Response 是一个请求级别的对象，继承自 Koa.Response。封装了 Node.js 原生的 HTTP Response 对象，提供了一系列辅助方法设置 HTTP 响应。

常用属性/方法

  1. response.status：设置响应状态码。
  2. response.body：设置响应体。
  3. response.set：设置响应头。

Controller

Controller 是控制器基类，所有的控制器都需要继承自 Controller。控制器负责处理具体的业务逻辑，调用 Service 或其他模块完成请求处理。

主要作用

  1. 处理请求：解析请求参数，调用 Service 处理业务逻辑。
  2. 返回响应：设置响应状态码、响应体等。

使用场景

  1. 处理具体的业务逻辑。
  2. 调用 Service 完成数据操作。

Service

Service 是服务基类，所有的服务都需要继承自 Service。Service 负责封装业务逻辑，供 Controller 调用。

主要作用

  1. 封装业务逻辑：将复杂的业务逻辑封装在 Service 中，保持 Controller 的简洁。
  2. 复用代码：多个 Controller 可以共享同一个 Service。

使用场景

  1. 数据库操作。
  2. 第三方 API 调用。
  3. 复杂的业务逻辑处理。

Helper

Helper 是工具类对象，提供一些实用的工具方法，可以在 Controller、Service、Middleware 中使用。

主要作用

  1. 提供工具方法：如生成随机数、格式化日期等。
  2. 复用工具逻辑：避免重复编写工具代码。

使用场景

  1. 格式化数据。
  2. 生成随机字符串或 ID。

Config

Egg.js 遵循配置和代码分离的原则，将一些需要硬编码的业务配置都放到配置文件中。同时，配置文件支持各个不同的运行环境使用不同的配置，所有框架、插件和应用级别的配置都可以通过 Config 对象获取到。

Logger

Logger 是日志对象，用于记录应用运行时的日志信息。Egg.js 提供了多种日志级别，如 debug、info、warn、error 等。

主要作用

  1. 记录日志：记录应用的运行状态、错误信息等。
  2. 调试应用：通过日志定位问题。

使用场景

  1. 记录请求日志。
  2. 记录错误日志。
  3. 记录调试信息。
